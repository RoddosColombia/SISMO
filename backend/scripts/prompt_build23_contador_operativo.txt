CORRECCIÓN CRÍTICA (ROG-1 violation identificada):
En backend/routers/ai_chat.py, la función crear_causacion o equivalente usa
service.request() para POST a Alegra. Debe usar request_with_verify() que hace
POST + GET de verificación. Sin esto el agente reporta éxito sin verificar HTTP 200.

   El patrón correcto es:

     resultado = await alegra_service.request("journals", "POST", body=payload)

           propone asiento DÉBITO/CRÉDITO al usuario →

SPRINT S2 — Facturación Ventas Motos F6: 1/10 → 8/10
════════════════════════════════════════════════════════════════

PROBLEMA ACTUAL:
POST /api/ventas/crear-factura existe pero no crea la factura en Alegra correctamente.
VIN y motor no aparecen en el ítem de Alegra con el formato requerido.
El inventario no se actualiza al crear la factura.
El loanbook pendiente_entrega no se crea.

QUÉ IMPLEMENTAR:

1. Endpoint POST /api/ventas/crear-factura en backend/routers/ventas.py:
   Payload requerido:
   {
     "cliente_nombre": str,
     "cliente_cedula": str,
     "cliente_telefono": str,
     "moto_vin": str,       # OBLIGATORIO
     "moto_motor": str,     # OBLIGATORIO
     "moto_modelo": str,
     "moto_color": str,
     "plan": str,           # P39S / P52S / P78S
     "modo_pago": str,      # semanal / quincenal / mensual
     "cuota_inicial": float,
     "precio_venta": float
   }

2. Formato OBLIGATORIO del ítem en Alegra (sin este formato el inventario no se actualiza):
   name: "[Modelo] [Color] - VIN: [chasis] / Motor: [motor]"
   Ejemplo: "TVS Raider 125 Negro - VIN: 9FL25AF31VDB95058 / Motor: BF3AT18C2356"
   price: precio_venta

3. Flujo al confirmar venta:
   a. Verificar moto en inventario_motos con estado == "Disponible" — si no, ERROR
   b. Crear cliente en Alegra si no existe (POST /contacts)
   c. Crear factura en Alegra con request_with_verify() POST /invoices
   d. Verificar HTTP 200 y obtener ID de factura
   e. Actualizar inventario_motos: estado → "Vendida", factura_alegra_id = ID
   f. Crear loanbook con estado "pendiente_entrega", cuotas=[], fecha_entrega=null
   g. Publicar evento factura.venta.creada al bus roddos_events
   h. Retornar {success: true, factura_id, loanbook_id}

4. Si falta VIN o motor → ERROR 400 antes de tocar Alegra. Nunca crear factura parcial.

5. Si Alegra falla → NO actualizar inventario ni crear loanbook (atomicidad).

SPRINT S5 — MÓDULO BACKLOG DE MOVIMIENTOS (NUEVO)
Pantalla separada en el menú de SISMO
════════════════════════════════════════════════════════════════

CONTEXTO:
Durante la conciliación bancaria, los movimientos con confianza <70% no se causan
automáticamente — van a un backlog. Actualmente ese backlog no tiene interfaz en SISMO.
Este módulo es el prerequisito para completar FASE 3 (subir extractos enero-marzo).

QUÉ IMPLEMENTAR — BACKEND:

1. Colección MongoDB: contabilidad_pendientes
   Schema de cada documento:
   {
     "_id": ObjectId,
     "banco": str,              # "bbva" | "bancolombia" | "nequi" | "davivienda"
     "extracto": str,           # "bbva_enero_2026"
     "fecha": str,              # "2026-01-02" YYYY-MM-DD
     "descripcion": str,        # texto original del banco
     "monto": float,            # negativo = egreso, positivo = ingreso
     "tipo": str,               # "EGRESO" | "INGRESO"
     "confianza_motor": float,  # 0.0 - 1.0
     "cuenta_sugerida": int,    # ID Alegra sugerido por motor (puede ser null)
     "razon_baja_confianza": str,
     "estado": str,             # "pendiente" | "causado" | "descartado"
     "journal_alegra_id": str,  # null hasta que se cause
     "resuelto_por": str,       # usuario que resolvió
     "creado_at": str,
     "resuelto_at": str
   }

2. Índice en contabilidad_pendientes: {estado: 1, banco: 1, fecha: 1}

3. Endpoints en backend/routers/backlog.py:
   
   GET /api/backlog/movimientos
   Query params: estado (default "pendiente"), banco, mes (YYYY-MM)
   Retorna lista paginada de movimientos pendientes con total_count
   
   POST /api/backlog/movimientos
   Crea un movimiento pendiente (usado por bank_reconciliation al encontrar <70%)
   Payload: banco, extracto, fecha, descripcion, monto, tipo, confianza_motor,
            cuenta_sugerida, razon_baja_confianza
   Anti-duplicado: hash(banco + fecha + descripcion + monto) — no crear si ya existe
   
   PATCH /api/backlog/movimientos/{id}/causar
   Payload: {cuenta_debito: int, cuenta_credito: int, observaciones: str}
   Crea el journal en Alegra con request_with_verify()
   Al éxito: estado → "causado", journal_alegra_id = ID retornado
   
   PATCH /api/backlog/movimientos/{id}/descartar
   Payload: {razon: str}
   Estado → "descartado"
   
   GET /api/backlog/stats
   Retorna: {total_pendientes, total_causados, total_descartados, por_banco: {...}}

4. El endpoint GET /api/backlog/stats también expone el conteo para el badge del menú.
