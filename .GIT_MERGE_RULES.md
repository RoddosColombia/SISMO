# Git Merge Rules para SISMO

## Regla Principal: Merges desde Worktrees a Main

Cuando hagas merge de una rama worktree (e.g., `merge/mora-2000-rule`, `claude/condescending-leakey`) a `main`:

### ✅ PROCEDIMIENTO CORRECTO

```bash
# 1. Asegurar que estás en main
git checkout main
git pull

# 2. Merge con --no-ff (crea commit de merge explícito)
git merge <rama> --no-ff

# 3. Si hay conflictos en archivos de configuración (.claude/, etc):
git checkout --ours .claude/  # Mantener main
git checkout --ours .

# 4. REVISAR MANUALMENTE:
# - backend/ (cambios de lógica)
# - frontend/ (cambios de UI)
# - render.yaml, requirements.txt

# 5. Resolver conflictos y agregar
git add .
git commit -m "Merge <rama> — [descripción de cambios]"
git push
```

### ❌ PROHIBIDO

- ❌ `git merge <rama> --ours` — Sobrescribe todo
- ❌ `git merge <rama> --squash` sin revisar — Pierde historial
- ❌ `git push` con conflictos sin resolver
- ❌ Hacer push directamente sin revisar backend/ y frontend/

## Estrategia de Conflictos

### Por archivo tipo:

| Archivo | Conflicto | Estrategia |
|---------|-----------|-----------|
| `.claude/settings.local.json` | Combinado | Usar `--ours` (main) |
| `backend/` | Lógica | Revisar CADA línea manualmente |
| `frontend/` | UI/componentes | Revisar CADA línea manualmente |
| `render.yaml` | Configuración | Usar `--ours` (main) |
| `requirements.txt` | Deps | Usar `--ours` (main) |
| `CATALOGO_DEFAULT` | Datos | **Revisar SIEMPRE — puede tener cambios críticos de negocio** |
| Funciones helper (mora, etc) | Lógica nueva | Restaurar si fue sobrescrito |

## Checklist Pre-Push

- [ ] `git status` — Sin conflictos sin resolver
- [ ] `git diff --cached` — Cambios visto y validado
- [ ] Backend: Endpoints existen y tienen lógica correcta
- [ ] Frontend: Componentes y funciones intactos
- [ ] MongoDB data: CATALOGO_DEFAULT, configuraciones críticas presentes
- [ ] Tests: Ejecutar `smoke_test_build23.py` si aplica
- [ ] Render health: `GET https://sismo-backend-40ca.onrender.com/api/health`

## Casos Especiales

### Caso 1: Cambios críticos de negocio (mora, multiplicadores, etc)

1. Hacer merge normal `git merge <rama> --no-ff`
2. Revisar MUY cuidadosamente backend/routers/loanbook.py
3. Si falta alguna función helper, restaurarla de git history:
   ```bash
   git show <commit-anterior>:backend/routers/loanbook.py | grep -A 50 "def calcular_mora" > /tmp/mora.py
   # Integrar manualmente
   ```
4. Verificar en MongoDB que catalogo_planes tiene todos los datos

### Caso 2: Múltiples merges con el mismo archivo

Si `backend/routers/loanbook.py` está en conflicto en múltiples merges:
1. No hacer `--ours` ciegamente
2. Usar `git diff <rama>..main -- backend/routers/loanbook.py`
3. Cherry-pick cambios específicos que se necesiten
